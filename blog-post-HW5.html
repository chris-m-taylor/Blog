4.5) Using the technique suggested here, where natural language descriptions are presented in a standard format, write plausible user requirements for the following functions:

An unattended gas pump system that includes a credit card reader. The customer swipes the card through the reader, then specifies the amount of fuel required. The fuel is delivered, and the customer's account debited.
  • The user will be able to choose either debit or credit
  • If the user chooses debit, the system will ask for a pin number, if the user chooses credit, the system will ask for a zip code.
  • There will be a $1.00 charge to the card to make sure the card is active before allowing for fuel to be delivered.
  • After fuel is delivered the $1.00 charge is refunded and the total amount of gas will be charged to the account.
The cash-dispensing function in a bank ATM.
  • The user will be able to insert their card to see their account information.
  • The user will not be allowed to remove the cash dispensed until their card is removed.
  • The ATM will dispense bills of size 20 only.
In an internet banking system, a facility that allows customers to transfer funds from one account held with the bank to another account with the same bank.
  • The user must specify the day they want their money to be sent.
  • The user must specify an amount and account they want to withdraw or deposit to.
  • The system will check for available funds based on the amount the user specified.
  • The money will be taken from users account immediately even if the funds to not transfer immediately.

4.6) Suggest how an engineer responsible for drawing up a system requirements specification might keep track of the relationships between functional and non-functional requirements.

A non-functional requirement is a constraint on the system as a whole while a functional requirement is how the system should react to certain inputs and how it will behave in different situations. An engineer may create a system requirements document that spells out what the system will accomplish at a high level in the different aspects of the project. A programmer will then take those requirements and expand upon the implementation in order to translate them to a coding language.

4.7) Using your knowledge of how an ATM is used, develop a set of use cases that could serve as a basis for understanding the requirements for an ATM system.

Use case for checking balance: The user approaches the atm and inserts their card --> atm then asks for a pin --> user then types in their pin --> atm displays two options, balance/withdraw --> user selects balance --> atm displays balance --> user removes card

Use case for withdrawing cash: The user approaches the atm and inserts their card --> atm then asks for a pin --> user then types in their pin --> atm displays two options, balance/withdraw --> user selects withdraw --> atm displays an amount that starts at $20 and goes up or down $20 at a time --> user clicks the up button 4 times to select $100 --> user clicks withdraw --> atm dispenses money but keeps it inaccessible to the user --> user removes card --> atm makes money accessible to user and logs out of their account.

Use case for incorrect pin: The user approaches the atm and inserts their card --> atm then asks for a pin --> user then types in their pin incorrectly --> atm displays “Wrong pin” --> user then types in their pin incorrectly --> atm displays “Wrong pin” --> user then types in their pin incorrectly --> atm displays “Call bank to unlock card” and no longer accepts card

Reflection on Chapter 4 and Readings

All of the articles and chapter 4 seemed to affirm the point that testing, and designing is not a one and done process. It is important to constantly revise the system throughout the entire process. Based on the article “Magical Number 7”, a human can only store 7 plus or minus 2 things at a time in their memory. This means that if we are not constantly testing and improving our system, we will not create a good system. In a case study that looked at tire pressure monitoring systems, they found that they could easily hack into these systems from 40m away. The manufacture assumed that since the network was behind the tire, it would eliminate the threat of someone tapping into the wireless network. They obviously were not testing and changing the product throughout the production. There is a process called test driven development (TDD) that is a mindset when creating a piece of software. The basic idea is to design a test and expect it to fail, then create the code to make the test pass. This idea allows every piece of code to be tested while creating to reduce the number of bugs in the software. Testing must be integrated into the core of our applications and TDD is a great way of solving some of the problems that have happened in the past.
